The shift

GitHub stops being just a warehouse for your files.

It becomes a governed boundary. Nothing crosses it in the clear, and what's stored there is inert on its own — you no longer have to trust the place that holds your data.

Before — a file warehouse you must trust. Anyone with access reads everything; once it's downloaded, it's gone.
After — a controlled gateway. Files enter only sealed; reading needs a live, authorized decision; you can revoke even what's already cloned.

How it works

01
🚦
The gateA required check blocks any plaintext data file from entering. It can't be bypassed — not even by an admin.
02
🔒
SealEach data file is sealed into a .ndc container. The plaintext is never inside it.
03
🗄️
StoreGitHub keeps the sealed bytes — like any file. It never holds a key and never decrypts. It stays simple.
04
✅
OpenTo read, an authorized person or process asks to open it. The decision lives with you, and every open is recorded.

What you get

🔑
Possession ≠ accessA developer can clone every file. Without authorization the bytes are useless. Downloading doesn't help them.
⚙️
Code keeps workingSource code stays normal and compiles. Authorized builds and apps open the data they need, transparently.
↩️
Revoke & proveCut access at any time — even to a file already cloned. Burn it everywhere. Every decision is a verifiable receipt.

Honest about the edges

  • At the moment someone is authorized to open a file, they can read it — we shrink the blast radius (revoke, audit, burn), we don't pretend to un-see an authorized read.
  • Instant revoke vs. offline speed is a dial you set.
  • The same idea protects any place you choose — not only GitHub: buckets, drives, folders.
